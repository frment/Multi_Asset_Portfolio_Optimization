# Multi-Asset Portfolio Optimization with Crypto Sleeve

Quantitative research project focused on risk-based portfolio construction, realistic constraints, out-of-sample validation, and conservative interpretation of crypto allocation.

## Research Question

Can a bounded allocation to BTC and ETH improve a traditional multi-asset portfolio after applying realistic constraints, turnover and cost frictions, robustness checks, tail-risk objectives, regime diagnostics, and a conservative supervised risk overlay?

## Current Project Status

The pipeline is implemented through Chapter 5 and structured as reproducible research:

1. Chapter 1: baseline walk-forward out-of-sample backtest.
2. Chapter 2: robustness, costs, and statistical confidence.
3. Chapter 3: tail-risk extension with historical CVaR optimization.
4. Chapter 4: regime diagnostics (in-sample attribution, not forecasting).
5. Chapter 5: supervised risk forecasting and conservative overlay evaluation.

Main research notebooks:

1. notebooks/01_backtest_analysis.ipynb
2. notebooks/02_robustness_analysis.ipynb

## Methodological Baseline

Current baseline conventions enforced in code and tests:

1. Calendar policy: business-day aligned.
2. Holding return engine: drifted buy-and-hold.
3. Rebalance policy: first available date each month on aligned index.
4. Annualization factor read from dataset metadata (baseline 252).
5. Regime reproducibility hardened with explicit dependency behavior.

This avoids common distortions from mixing 24/7 crypto with weekday ETF calendars using naive calendar-day assumptions.

## Design Principles

1. **Modularity**

   * Each concern has its own module: data, preprocessing, metrics, optimization, backtesting, costs, robustness and tail risk.

2. **Config-driven research**

   * Parameters live in YAML rather than being scattered through notebooks.

3. **Pure functions first**

   * Prefer small, testable functions over hidden notebook state.

4. **Walk-forward discipline**

   * Portfolio weights are estimated using historical information only.

5. **Benchmark discipline**

   * Every optimizer is compared against simple baselines and controls.

6. **Risk-first interpretation**

   * Sharpe is not enough. Drawdown, Expected Shortfall, turnover and crypto exposure matter.

7. **Honest conclusions**

   * The project documents fragility, limitations and negative evidence.

## Important Limitations and Disclaimer

* This is a research and learning project. It is **not investment advice**.
* Results are based on historical data and may not generalize.
* Crypto has limited history and materially different risk characteristics from traditional assets.
* Data comes primarily from Yahoo Finance and is not institutional-grade.
* Transaction costs are currently modeled as simple proportional turnover costs.
* Slippage, market impact, borrow costs, taxes and liquidity constraints are not fully modeled.
* CVaR is estimated empirically from historical windows; no EVT or parametric tail model is used yet.
* Stress windows are diagnostic and historical, not predictive.
* Regime analysis is planned next but not yet implemented.
* This remains a research codebase, not a production portfolio management system.
